## 1. Обзор продукта

**AutoHelper** — SaaS-платформа для учёта автомобилей и их истории обслуживания. Платформа обеспечивает прозрачность технического состояния автомобиля для любого пользователя, предоставляет AI-ассистента для диагностики и консультаций, а также объединяет клиентов с партнёрами (автосервисы, автомойки, эвакуаторы, магазины автозапчастей и прочие).

---

## 2. Роли пользователей

| Роль | Описание |
|------|----------|
| **Клиент** | Зарегистрированный пользователь. Может владеть автомобилями, управлять их статусами, пользоваться AI-чатом по подписке |
| **Партнёр** | Владелец бизнеса (автосервис, автомойка, эвакуатор и др.). Имеет отдельную учётную запись и партнёрский кабинет. Размещает рекламу и получает заявки от клиентов через AI-ассистента |
| **Администратор** | Сотрудник платформы. Управляет всеми сущностями через отдельную панель |

---

## 3. Модуль: Управление клиентами

### 3.1 Регистрация и аутентификация

- Клиент регистрируется по email и паролю
- Аутентификация реализована через JWT (access + refresh токены)
- Поддерживается смена пароля и загрузка аватара

### 3.2 Профиль клиента

Профиль содержит: имя, контактные данные, email, статус подписки, дата регистрации.

Клиент может оставить отзыв о платформе AutoHelper через личный кабинет (см. раздел 11).

### 3.3 Публичный профиль

Любой авторизованный пользователь может найти клиента по VIN его автомобиля и увидеть публичную информацию о владельце.

## 4. Модуль: Управление автомобилями

### 4.1 Реестр автомобилей

- Каждый автомобиль регистрируется **один раз** в системе
- **VIN-номер уникален** — система не допускает создание двух автомобилей с одинаковым VIN
- Любой клиент может найти и просмотреть любой автомобиль в системе по VIN
- При просмотре автомобиля отображается информация о текущем владельце

**Атрибуты автомобиля:**

| Поле | Описание |
|------|----------|
| VIN | Уникальный идентификатор (обязательное) |
| Марка | Производитель |
| Модель | Модель автомобиля |
| Год выпуска | |
| Цвет | |
| Пробег | Актуальный пробег |
| Статус | Текущий статус автомобиля |
| Владелец | Привязка к клиенту |

### 4.2 Статусы автомобиля

Владелец автомобиля может выставить один из следующих статусов:

| Статус | Дополнительные данные | Требования |
|--------|----------------------|------------|
| **Активен** | — | По умолчанию |
| **На продаже** | — | Автомобиль отображается в публичном каталоге продаж |
| **В ремонте** | Наименование сервиса (партнёра) | Обязательно указать, где проводится ремонт |
| **Утилизирован** | Подтверждающий документ (PDF) | Обязательна загрузка документа |
| **Разобран** | Подтверждающий документ (PDF) | Обязательна загрузка документа |

## 5. Модуль: История технических работ (Service Records)

### 5.1 Записи о работах

К каждому автомобилю привязывается история выполненных технических работ. Записи **публично доступны** — любой пользователь может просмотреть историю работ по VIN.

**Атрибуты записи о работе:**

| Поле | Описание |
|------|----------|
| Название работы | Краткое наименование |
| Описание | Подробное описание выполненных операций |
| Дата выполнения | |
| Стоимость | |
| Исполнитель | Партнёр или сторонний сервис, вместе с контактными данными |
| Перечень операций | Список конкретных выполненных действий |
| Документ-наряд | PDF-файл, подтверждающий выполнение работ |

### 5.2 Документ-наряд (PDF)

- Каждая запись о работе **должна содержать** PDF-документ (наряд-заказ)
- Документ загружается при создании записи
- Документ доступен для скачивания из карточки работы
- Документ можно менять на другой, но при этом в системе хранения документ ещё будет существовать
- Каждое изменение документа-наряда должно логироваться в БД
- Хранение файлов — в object storage (S3 / MinIO)
- **Загружаемые файлы сжимаются** без потери качества (либо применяется иная стратегия рационального использования размеров документов)

## 6. Сквозные правила работы с данными

### 6.1 Soft-delete

**Все удаления сущностей в БД являются мягкими (soft-delete).** Физическое удаление записи из БД не производится. Удалённые сущности помечаются флагом `IsDeleted = true` и исключаются из запросов через глобальные фильтры EF Core.

Затрагивает: Клиенты, Автомобили, Записи о работах, Партнёры, Отзывы, Рекламные кампании и все прочие доменные сущности.

### 6.2 Аудит-лог изменений (AuditLog)

Все изменения (создание, редактирование, удаление) клиента, автомобиля и записей о выполненных работах **логируются в единой таблице `AuditLogs`**.

**Структура записи аудит-лога:**

| Поле | Тип | Описание |
|------|-----|----------|
| `Id` | Guid | Идентификатор записи |
| `OperationType` | Enum | `Created` / `Updated` / `Deleted` |
| `EntityType` | string | Тип сущности (`Customer`, `Vehicle`, `ServiceRecord`) |
| `EntityId` | Guid | ID изменённой сущности |
| `PerformedAt` | DateTime (UTC) | Время операции по UTC |
| `PerformedByUserId` | Guid? | ID клиента или администратора, выполнившего операцию |
| `PerformedByRole` | string | Роль: `Client` / `Admin` / `System` |
| `AdditionalInfo` | string? | JSON-строка с дополнительной информацией |

**Заполнение `AdditionalInfo`:**
- Для операции `Updated` по сущности `ServiceRecord`: JSON-объект вида `{"OldEntity": "<json-представление старой сущности>"}`.
- Для остальных сущностей и операций: поле может быть пустым или расширяться по мере необходимости.

## 7. Модуль: AI АвтоПомощник

### 7.1 Общее описание

**АвтоПомощник** — AI-чат-ассистент, доступный клиенту **по платной подписке**. Ассистент работает на базе LLM и специализируется исключительно на автомобильной тематике.

**Ключевые требования к реализации:**
- АвтоПомощник отвечает **на языке текущей локализации сайта** (ru / en)
- Бэкенд реализуется через **абстракцию LLM-провайдера** (`ILlmProvider`), что позволяет переключить провайдера/модель без изменения бизнес-логики. Начальный провайдер — **OpenAI Responses API**.
- **Защита от prompt injection**: свободный ввод пользователя держится отдельно от системных инструкций; форм-поля используют enum/select где возможно; backend-факты передаются отдельным структурированным блоком; вся бизнес-логика принимает только schema-validated JSON.
- Бот **не должен выдавать выдуманные ответы и факты**, не должен давать незаконные советы.
- Все ответы LLM, используемые бизнес-логикой и UI, возвращаются в формате **Structured Outputs** по заранее определённым JSON Schema.

> **Ограничение тематики:** АвтоПомощник отвечает **только** на вопросы, относящиеся к трём режимам (см. ниже). Вопросы вне этих тем (технические детали реализации, другие сферы деятельности и т.д.) отклоняются с вежливым сообщением.

### 7.2 Архитектура оркестратора

Вся логика обработки сообщения реализована в виде **`AutoAssistantOrchestrator`** — сервиса Application-слоя, который последовательно вызывает следующие компоненты:

| Компонент | Описание |
|-----------|----------|
| `RequestClassifier` | Маршрутизация и валидация запроса (модель `gpt-5.4-nano`) |
| `ContextAssembler` | Сбор детерминированных данных: клиент, авто, история работ, summary предыдущих чатов |
| `BackendDataGateway` | Получение данных из БД и внешних API (партнёры, ценовые бенчмарки, Google Places) |
| `ResponseGenerator` | Генерация структурированного ответа (модель `gpt-5.4-mini`, эскалация на `gpt-5.4`) |
| `ChatStateManager` | Управление состоянием чата (active / awaiting_user_answers / final_answer_sent / completed) |
| `InvalidRequestLogger` | Логирование невалидных запросов в `InvalidChatRequests` |
| `UsageMeter` | Учёт использованных запросов по подписке |

**LLM не занимается самостоятельным поиском фактов.** LLM должна: классифицировать запрос, определить режим, запросить недостающие данные, интерпретировать уже собранные backend-данные, сформировать структурированный ответ для UI.

#### Выбор модели

| Назначение | Модель |
|-----------|--------|
| Маршрутизация, валидация, классификация, определение недостающих полей | `gpt-5.4-nano` |
| Основной ответ пользователю (диагностика, анализ работ, оформление поиска партнёров) | `gpt-5.4-mini` |
| Эскалация для сложных случаев (много конфликтующих признаков, длинная история, высокий риск ошибки) | `gpt-5.4` |

#### Интерфейс ILlmProvider

```csharp
public interface ILlmProvider
{
    Task<LlmJsonResult<T>> GenerateStructuredAsync<T>(
        string model, string systemPrompt, object userInput, object jsonSchema, CancellationToken ct = default);
    Task<LlmTextResult> GenerateTextAsync(
        string model, string systemPrompt, object userInput, CancellationToken ct = default);
    Task<string> SummarizeConversationAsync(
        string model, IReadOnlyCollection<ChatMessage> messages, CancellationToken ct = default);
}
```

`ILlmProvider` — тонкая абстракция над провайдером. Вся бизнес-логика — в оркестраторе и сервисах.

### 7.3 Инструментарий на странице АвтоПомощника

На странице чатбота должны быть **визуально отображены инструменты** (режимы работы) как элементы чата. Клиент выбирает нужный инструмент/режим прямо в интерфейсе чата, после чего бот переключается в соответствующий режим и предоставляет соответствующую форму или интерфейс.

### 7.4 Привязка чата

- Чат **всегда** привязан к клиенту (история сохраняется)
- Чат **опционально** может быть привязан к конкретному автомобилю клиента
- При привязке к автомобилю — данные авто (марка, модель, год, пробег) и история работ автоматически передаются в контекст LLM
- Для длинных чатов применяется **сжатие/summarization** истории (через `SummarizeConversationAsync`)

### 7.5 Режимы работы АвтоПомощника

#### Режим 1: Помощь с неисправностями и советы (`diagnostics`)

**Триггер:** клиент описывает проблему в процессе эксплуатации автомобиля.

**Многошаговый сценарий:**

1. Клиент заполняет **стартовую форму**:
   - Выбор автомобиля (обязательно)
   - Описание симптомов
   - Последние подозрительные действия/ситуации
   - Ранее возникавшие неполадки

2. После первичной оценки бот может запросить **уточнения** — отдельными сообщениями в чате (не одним абзацем, а по одному вопросу на сообщение). Таких уточнений может быть несколько.

3. После получения ответов backend повторно собирает контекст и отправляет запрос в LLM — получает либо ещё список уточнений, либо **итоговый диагноз**.

4. **Итоговый ответ** — в структурированной форме:
   - Список потенциальных проблем с **вероятностью** каждой (0–1)
   - Возможные причины
   - Рекомендуемые действия
   - Текущие риски
   - Уровень срочности (`low` / `medium` / `high` / `stop_driving`)
   - Безопасность продолжения езды
   - Предложения партнёров (категория сервиса + срочность) — если включено в конфигурации
   - Дисклеймер об оценочном характере ответа

5. После **итогового ответа** бот разрешает **ровно 1 дополнительный вопрос** клиента по сформированному ответу. После ответа на этот вопрос чат переходит в состояние `completed` (только чтение).

**Состояния чата в этом режиме:**
- `active` → `awaiting_user_answers` → `active` → ... → `final_answer_sent` → `completed`

**Невалидный запрос:** нерелевантная тематике информация — бот сообщает о невалидности, счётчик запросов **не уменьшается**, запрос фиксируется в `InvalidChatRequests`.

**Администрирование:** В Admin-панели предусмотрена возможность **отключить предложения партнёров** в этом режиме.

#### Режим 2: Уточнения за выполненные работы (`work_review`)

**Триггер:** клиент спрашивает про работы, которые ему предложили или уже выполнили в сервисе.

**Поведение ассистента:**
1. Предоставляет клиенту **форму с полями**:
   - Перечень выполненных работ
   - Использованные детали и под каким предлогом
   - Причины, по которым предложили выполнить работы
   - Стоимость работ и деталей
   - Гарантии и обещания, данные при выполнении работ

2. Backend предварительно получает **рыночные бенчмарки по ценам**, затем модель объясняет клиенту результат. Ответ в **структурированной форме**:
   - Насколько актуальны предлоги для выполнения работ (`low/medium/high/unclear`)
   - Сравнение цены работ со средней рыночной (`below_market/near_market/above_market/unknown`)
   - Сравнение цены деталей со средней рыночной
   - Оценка гарантий (`weak/normal/strong/unclear`)
   - Общая оценка честности сервиса (`poor/mixed/fair/good/unknown`)
   - Ожидания от дальнейшего обслуживания
   - Через какой километраж потребуется повторная замена

3. **Невалидный запрос:** нерелевантная информация — бот сообщает о невалидности, запрос фиксируется в `InvalidChatRequests`.

#### Режим 3: Советы и предложения партнёров (`partner_search`)

**Доступность:** требует активной подписки, но **не расходует лимит запросов** — доступен неограниченно.

**Триггер:** клиент описывает ситуацию, требующую услуг (например: пробитое колесо, нужна эвакуация, поиск сервиса).

**Ключевой принцип: LLM не занимается самостоятельным поиском мест.** LLM только:
1. Определяет **категорию услуги** (`tow_truck`, `tire_service`, `car_service`, `electrician` и др.)
2. Оформляет **итоговый ответ** на основе карточек, уже подготовленных backend

**Backend-логика поиска партнёров:**
1. Если геолокация клиента не предоставлена — система запрашивает её
2. Backend загружает **активных собственных партнёров** из БД по категории + геолокации
3. Лимит выдачи `N` задаётся в Admin-панели (3–5)
4. Если собственных партнёров **≥ N** — внешние вызовы не делаются; список формируется только из собственных
5. Если собственных партнёров **< N** — backend дополняет до лимита через **Google Places API** (Nearby/Text Search + Place Details только для нужного количества кандидатов)
6. Собственные партнёры **всегда идут первыми** и могут вытеснить Google-кандидатов из финального топа
7. LLM получает готовые карточки и формирует понятный ответ для пользователя

**Карточка партнёра** содержит: источник (собственный / Google Places), рейтинг, количество отзывов, открыт ли сейчас, расстояние, услуги, предупреждающий флаг, адрес, телефон, сайт.

**Предупреждение** отображается, если партнёр отмечен как потенциально профнепригодный.

### 7.7 История чатов

Клиент имеет доступ ко всей истории своих чатов с АвтоПомощником, включая возможность просматривать переписку. Завершённые (`completed`) чаты доступны **только для чтения**.

### 7.8 Подписки на АвтоПомощника

Доступ к чатботу (все три режима) предоставляется **только по активной платной подписке**. Без подписки чатбот недоступен.

**Механика расхода запросов:**
- **Режим 1 (diagnostics)** и **Режим 2 (work_review)** — тратят запросы из лимита подписки
- **Режим 3 (partner_search)** — **не тратит запросы**, доступен неограниченно в рамках активной подписки

#### Тарифные планы чатбота

| План | Цена | Запросы (Режим 1 + Режим 2 суммарно) | Режим 3 |
|------|------|--------------------------------------|---------|
| **Обычный** | $4.99 / месяц | 10 запросов в месяц | Без ограничений |
| **Про** | $7.99 / месяц | 20 запросов в месяц | Без ограничений |
| **Максимум** | $12.99 / месяц | 40 запросов в месяц | Без ограничений |

- На странице чатбота клиенту **отображается счётчик**: сколько запросов (Режим 1 + 2) осталось и через сколько дней они обновятся
- Если запросы исчерпаны — клиент может **докупить 10 запросов за $3** (разово, применяется только к Режимам 1 и 2)

#### Конфигурация в Admin-панели

| Параметр | Описание |
|----------|----------|
| Включить / отключить чатбот | При отключении — на странице чатбота отображается «Технические работы» |
| Максимальное количество символов в полях форм | Глобально или для каждого поля отдельно |
| Максимальное количество запросов в день для каждого типа подписки | |
| Цены подписок и количество запросов в них | Обычный / Про / Максимум |
| Цена и количество запросов при разовом пополнении | По умолчанию: $3 / 10 запросов |

### 7.9 Невалидные запросы к чатботу

Все невалидные запросы (нерелевантные тематике выбранного режима) фиксируются в таблице `InvalidChatRequests`:

| Поле | Описание |
|------|----------|
| `CustomerId` | ID клиента |
| `InvalidAttemptsCount` | Суммарное количество невалидных попыток |
| `LastInvalidAt` | Дата/время последней невалидной попытки |
| `Details` | JSON-массив с деталями каждого невалидного запроса (текст запроса, режим, время) |

- Количество невалидных запросов по клиентам **отображается в Admin-панели**

## 8. Модуль: Партнёры и интеграции

### 8.1 Типы партнёров

- Автомагазины (запчасти, аксессуары)

### 8.2 Учётная запись партнёра

Для каждого партнёра создаётся **отдельная учётная запись** со своими правами:
- Партнёр может входить **только в свой партнёрский кабинет**
- Партнёрам **не показываются рекламные баннеры**
- Партнёр **не имеет доступа** к функциям клиентского раздела

### 8.3 Партнёрский кабинет

Партнёр может управлять своим профилем и рекламой:
- Просмотр и редактирование профиля партнёра
- **Заявки администраторам на оплату показа рекламных баннеров** по часам и/или дням
- Настройка: показывать ли рекламные баннеры **анонимным пользователям** (да / нет)

### 8.4 Профиль партнёра

**Атрибуты партнёра:**

| Поле | Описание |
|------|----------|
| Название | |
| Тип / категория услуг | Автосервис, автомойка, эвакуатор и т.д. |
| Специализация | Детальное описание предоставляемых услуг |
| Описание | Общая информация о компании |
| Адрес | |
| Геолокация | Координаты (lat/lng) для геопоиска |
| Время работы | Расписание по дням недели; поддержка режима «круглосуточно» |
| Контакты | Телефон, сайт, мессенджеры |
| Логотип | |
| Подтверждающие документы | Лицензии, сертификаты |

Новый партнёр проходит **верификацию администратором** перед публикацией.

### 8.5 Рейтинги и отзывы

#### Правила выставления оценок

Клиент может оставить оценку партнёру **только при наличии факта взаимодействия**:

- Партнёр был рекомендован клиенту через АвтоПомощника, **или**
- Партнёр указан исполнителем в записи о выполненных работах автомобиля клиента

Без одного из этих фактов оценку выставить невозможно.

#### Шкала оценок

- Оценка по шкале **1–5**
- Обязательный текстовый комментарий

#### Автоматическая пометка «Потенциально профнепригодный»

Партнёр автоматически получает метку **⚠️ Потенциально профнепригодный**, если:

> Количество оценок **ниже 3** составляет **5 или более**

**Последствия метки:**
- В АвтоПомощнике рядом с именем партнёра отображается предупреждение
- В административной панели партнёр выделяется в специальном разделе для проверки
- Администратор может принять решение о прекращении сотрудничества и деактивации рекламы

**Администрирование:** В Admin-панели предусмотрена возможность **редактировать и корректировать** оценки и отзывы партнёров.

### 8.6 Рекламные интеграции

Партнёры могут размещать рекламу на платформе в двух форматах:

| Формат | Описание |
|--------|----------|
| **Блок «Предложения»** | Тематический блок на главной и релевантных страницах |
| **Рекламные баннеры** | Баннеры на страницах сайта |

**Правила показа баннеров:**
- Рекламные баннеры **могут показываться анонимным пользователям** (если партнёр разрешил)
- **Партнёрам не показываются** рекламные баннеры других партнёров
- Реклама таргетируется по **категории услуг**
- АвтоПомощник также может **органично предлагать** партнёров в рамках Режима 1 и Режима 3

**Конфигурация рекламы в Admin-панели:**
- Настройка показа баннеров и блоков «Предложений» **по количеству** (максимум N баннеров одновременно)
- Настройка показа **по дням**
- Управление заявками партнёров на показ баннеров (по часам / дням)
- Ценообразование показов баннеров (цена за час / за день) — устанавливается администраторами

### 12.2 Инфраструктурные абстракции

Все инфраструктурные компоненты на бэкенде реализуются **через хорошие уровни абстракции** (интерфейсы в `Application`), что позволяет:
- Заменить провайдера файлового хранилища (S3/MinIO → любой другой) без изменения бизнес-логики
- Заменить LLM-провайдера (OpenAI → Anthropic / другой) без изменения логики чата
- Заменить платёжный шлюз (Lemon Squeezy → другой) без изменения логики биллинга

### 12.3 Сжатие файлов

При загрузке файлов (PDF, изображения):
- PDF-документы: рассматривается возможность сжатия без потери читаемости
- Изображения (аватары, логотипы): сжатие без потери видимого качества (например, WebP / настройка quality)
- Альтернатива: ограничение максимального размера загружаемых файлов с UX-уведомлением

### 12.4 Локализация фронтенда

Фронтенд **обязан поддерживать мультиязычность** с удобной структурой для добавления новых языков.

**Требования:**

- Библиотека локализации: **`next-intl`** (официально поддерживается Next.js App Router)
- Язык по умолчанию: **русский (ru)**
- Дополнительный язык: **английский (en)**
- Все текстовые строки UI (лейблы, кнопки, сообщения об ошибках, плейсхолдеры) хранятся **исключительно в файлах переводов** — никаких хардкодированных строк в компонентах
- Структура файлов переводов: `messages/ru.json`, `messages/en.json`
- Язык определяется автоматически по настройкам браузера пользователя и может быть изменён вручную через компонент `LocaleSwitcher`
- Выбранный язык сохраняется между сессиями (cookie)
- Все новые страницы и компоненты **обязаны** использовать хук `useTranslations` вместо хардкодированных строк
- **АвтоПомощник отвечает на языке текущей локализации сайта**

```bash

  --project src/AutoHelper.Infrastructure \
  --startup-project src/AutoHelper.Api \
  --output-dir Persistence/Migrations
```

## 13. Ключевые бизнес-правила

1. **Уникальность VIN** — в системе не может быть двух автомобилей с одинаковым VIN.
2. **Публичность истории** — история работ любого автомобиля доступна всем пользователям по VIN.
3. **Подтверждающий документ** — статусы «Утилизирован» и «Разобран» требуют обязательной загрузки PDF-документа.
4. **Право на оценку** — клиент может оценить партнёра только при наличии задокументированного факта взаимодействия.
5. **Порог профнепригодности** — 5 и более оценок ниже 3 автоматически помечают партнёра как потенциально профнепригодного.
6. **Ограничение тематики ИИ** — АвтоПомощник отвечает строго в рамках трёх режимов; прочие вопросы отклоняются.
7. **Подписка на ИИ** — доступ ко всем режимам АвтоПомощника требует активной платной подписки. Режимы 1 и 2 расходуют лимит запросов; Режим 3 не расходует запросы, но также требует подписки.
8. **Локализация UI** — все текстовые строки интерфейса хранятся в файлах переводов; хардкодированный текст в компонентах недопустим.
9. **Soft-delete** — физическое удаление сущностей из БД запрещено; все удаления — мягкие (IsDeleted = true).
10. **Аудит-лог** — все операции создания, изменения и удаления клиентов, автомобилей и записей о работах фиксируются в таблице AuditLogs.
11. **Невалидный запрос чатботу** — не засчитывается как потраченный запрос, фиксируется в таблице InvalidChatRequests.
12. **Партнёрский кабинет изолирован** — партнёры видят только свой кабинет, им не показываются рекламные баннеры.
13. **LLM-абстракция** — провайдер LLM подключается через интерфейс и может быть заменён без изменения бизнес-логики.
14. **Инфраструктурные абстракции** — все внешние сервисы (storage, LLM, billing) подключаются через интерфейсы Application-слоя.
